## 2. What just landed (D5.7b summary)

Five commits implemented the cross-pond import UX:

### 2.1 D5.7b.1 — verb split (`4cb992d4`)

`pond remote add NAME URL PATH` is **always pull-mode**. `PATH = /`
is a mirror restart (foreign `store_id` must equal local `pond_id`);
non-root PATH (e.g. `/imports/upstream`) is a cross-pond import
(foreign `store_id` must differ). `pond backup add NAME URL
[--bidirectional]` is push by default, push+pull with the flag.
The old `pond remote add NAME URL --mode=…` form was removed
outright (D5.7b never shipped to production, so no deprecation
window was needed).

### 2.2 D5.7b.2 — first-pull mount materialization (`7e763a42`)

When the first `pond pull <name>` runs for a pull-mode attachment
with a non-root `mount_path` and a foreign `store_id`, the consumer
materializes a directory entry at `mount_path` whose `pond_id`
differs from its parent's. Subsequent reads through that path
resolve into the foreign pond's rows naturally; no separate "import"
concept is needed at the row layer.

### 2.3 D5.7b.3 — read-only foreign mounts (`6632f1ea`)

Two enforcement points:

- `WD::insert_node` now calls `check_writable()` on the parent
  before delegating to `self.dref.insert`. `effective_root`
  propagation means any descendant of a foreign mount is also
  read-only (writes return `ReadOnlyImport`).
- `discover_post_commit_factories` in `crates/steward/src/guard.rs`
  filters its matches by local `pond_id`, so a factory living in a
  foreign pond's `/system/run/*` directory never auto-executes on
  the local pond.

`discover_sys_remotes` does NOT need the same filter (its glob
`/sys/remotes/*` is anchored at root and doesn't recurse into
`/imports/upstream/sys/remotes/*`).

### 2.4 D5.7b.4 — detach vs `--purge` (`2f95f99b`)

Default `pond remote remove NAME` is a **detach**: clear
`/sys/remotes/<name>` YAML, `remote_mode:<name>`,
`remote_mount_path:<name>`, and the per-URL watermarks. The
materialized mount entry is left in place so previously-imported
data stays readable.

With `--purge`: also unlink the mount entry from its parent
directory. Physically vacuuming the foreign pond_id's rows from the
underlying Delta log is **deferred** — the rows are unreachable by
path after purge and are eligible for a future compaction.

### 2.5 D5.7b.5 — attach-time conflict checks (`bfcfe76e`)

Two pre-attach validators in `add_remote_attachment_internal`
(pull-mode only):

1. `validate_no_mount_path_conflict` — refuses if another pull-mode
   remote already uses the same `mount_path` (with trailing-slash
   normalization). `--overwrite` only honors same-name re-attach,
   never cross-name conflicts.
2. `validate_no_foreign_store_id_collision` — refuses if a
   different pull-mode name already mounts the same foreign
   `store_id`. Mirror-restart attachments (`PATH=/`, same
   `store_id`) are exempt because they don't materialize a mount
   entry.

## 3. The cross-pond invariants you can now rely on

These were the design decisions locked in D5.7b. Tests can assume
them; new factories and CLI features must preserve them.

| # | Invariant | Enforced where |
|---|-----------|----------------|
| 1 | A foreign `pond_id` reachable through a mount path is **strictly read-only**. | `WD::insert_node::check_writable` |
| 2 | The steward's post-commit auto-exec only scans `/system/run/*` entries owned by the local `pond_id`. | `discover_post_commit_factories` filter |
| 3 | A mount path uniquely identifies a foreign pond inside the local pond. Two pull-mode attachments cannot share a mount path. | `validate_no_mount_path_conflict` |
| 4 | A foreign `store_id` is reachable through at most one mount path. Two pull-mode attachments cannot share a `store_id`. | `validate_no_foreign_store_id_collision` |
| 5 | Removal is non-destructive by default. `--purge` is the only way to drop the mount entry. | `remove_remote_command` |
| 6 | `pond push` filters by local `pond_id`. Foreign rows never leak back upstream. | D5.3 (`Remote::push` partition filter) |
| 7 | Mirror restart (`PATH=/`, same `store_id`) and cross-pond import (`PATH!=/`, different `store_id`) are mutually exclusive at attach time. | URL-probe match arms in `add_remote_attachment_internal` |

## 7. Quick reference — D5.7b CLI shapes

```bash
# Cross-pond import (foreign store_id must differ from local pond_id)
pond remote add upstream s3://prod-bucket /imports/upstream \
    --region us-east-1 --access-key-id ... --secret-access-key ...

# Mirror restart (foreign store_id must match local pond_id)
pond remote add origin file:///mnt/backups/origin /

# Push-only backup
pond backup add origin s3://my-bucket --region us-east-1 ...

# Bidirectional remote (push + pull)
pond backup add hub s3://hub-bucket --bidirectional ...

# Detach (keeps imported data readable by path)
pond remote remove upstream

# Purge (also removes the mount entry; foreign rows become unreachable)
pond remote remove --purge upstream
```

Attach-time refusals to expect when writing tests:

- `mount path \`X\` is already used by remote \`Y\`` — duplicate-path collision (§ 3 inv. 3).
- `foreign store_id \`X\` is already mounted by remote \`Y\`` — duplicate-store_id collision (§ 3 inv. 4).
- `mount path \`/\` is reserved for mirror restarts` — using `/` with a foreign `store_id`.
- `mount path \`X\` is reserved for cross-pond imports` — using a non-root path with the local `store_id`.
- `ReadOnlyImport` — any write inside `/imports/<name>/...`.
