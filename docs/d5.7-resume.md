## 2. D5.7a — Verify RECORDED side (the smaller, mechanical fix)

### What's wrong today

Every duckpond write transaction goes through this path in `crates/steward/src/control_table.rs:287-305`:

```rust
pub async fn record_data_committed(
    &mut self,
    txn_meta: &PondTxnMetadata,
    _transaction_type: TransactionType,
    data_fs_version: i64,
    duration_ms: i64,
) -> Result<(), StewardError> {
    let metadata = DataCommittedMetadata {
        partition_checksums: HashMap::new(),  // <-- LIES.
        data_delta_version: data_fs_version,
    };
    let metadata_json = serde_json::to_string(&metadata).unwrap_or_else(|_| "{}".into());
    // ... write the DataCommitted record ...
}
```

That `HashMap::new()` is the bug.  When `Remote::push` later serializes
this record into Checksum rows on the remote (see
`crates/sync-remote/src/remote.rs:1003`), it writes zero rows.  When a
consumer runs `verify_against_remote`, it gets:
- **live**: real per-partition Merkle checksums from D5.5.
- **recorded** (from the remote's Checksum rows): empty map.
- **mismatch report** for every partition the consumer has.

Cross-pond import via `apply_pulled_bundle` (in
`crates/steward/src/remote_adapter.rs:511-517`) copies the bundle's
`partition_checksums` straight into the consumer's local DataCommitted
record, so the consumer's "RECORDED" matches its "LIVE" only for data
it imported, never for data it wrote natively.

### How sync_steward does it (the reference)

The sync_steward sandbox computes checksums at commit time.  See
`crates/sync-steward/src/steward.rs:971-984`:

```rust
async fn snapshot_all_partition_checksums(&self) -> Result<PartitionChecksums> {
    let partitions = self.store.partitions(self.store_id).await?;
    let mut out: PartitionChecksums = HashMap::new();
    for p in partitions {
        let cs = self
            .store
            .compute_partition_checksum(self.store_id, &p, &*self.checksum_strategy)
            .await?;
        out.insert(p, cs);
    }
    Ok(out)
}
```

This runs inside the write transaction (the steward holds an exclusive
write lock on its k/v store) and reads from the IN-MEMORY uncommitted
state.  After the commit lands, the recorded checksums match exactly
what `compute_live_checksums` would return for the same `(pond_id,
seq)` snapshot.

### How duckpond needs to do it

Duckpond differs in two important ways from sync_steward:

1. **Row schema, not k/v.**  Every commit appends rows to a Delta
   table; there's no in-memory uncommitted state to query in the
   same way.  The "snapshot at commit time" needs to be the Delta
   table's state AFTER the commit lands (so the new rows are
   visible), but BEFORE control table records the DataCommitted row.
2. **The commit happens at the tlogfs layer, not in
   `record_data_committed`.**  `record_data_committed` runs LAST,
   after the data table commit has already landed.  See
   `crates/steward/src/guard.rs:300-339`:

   ```rust
   match commit_result {
       Ok(Some(())) => {
           let new_version = pre_commit_version.unwrap_or(0) + 1;
           // ...
           self.control_table
               .record_data_committed(
                   &self.txn_meta,
                   self.transaction_type,
                   new_version,     // <-- the Delta version that just landed
                   duration_ms,
               )
               .await
               // ...
       }
   }
   ```

So the strategy is:

1. **At `record_data_committed` call time, the data Delta commit has
   already landed.**  Reading via `compute_live_checksums(pond_id)`
   over the post-commit Delta state gives you exactly the checksums
   you want to record.
2. **The control table doesn't know about the Ship's data
   persistence.**  Today `record_data_committed` only has access to
   `self.inner` (the underlying `sync_steward::ControlTable`) and the
   metadata structs.  The data persistence layer lives in the Ship.
3. **Therefore, the call site needs to pass the snapshotted
   checksums INTO `record_data_committed`.**  Either:
   - **Option A (recommended)**: extend the signature to take a
     `PartitionChecksums` argument.  The caller in
     `crates/steward/src/guard.rs:300` computes them via
     `ShipRemoteSteward::compute_live_checksums(local_pond_id)` AFTER
     the data commit lands, BEFORE writing DataCommitted, and threads
     them through.
   - **Option B (worse)**: give `ControlTable` a back-reference to
     the Ship/data persistence layer.  Creates a circular dependency
     and breaks the layering.

Option A is the way.  See "Implementation sketch" below.

### Implementation sketch for D5.7a

### Subtleties for D5.7a

1. **TransactionGuard ownership.**  `commit()` in `StewardTransactionGuard`
   holds `&mut self` and the ship reference.  The new
   `ShipRemoteSteward::new(self.ship)` call needs `&mut Ship`.  Check
   that `self.ship` is reachable as `&mut` at that point (it should
   be — the data commit already happened).  If lifetimes complain,
   take the `&mut Ship` reference before calling
   `record_data_committed` and pass the snapshotted map by value.

2. **`compute_live_checksums` cost.**  D5.5's implementation runs a
   DataFusion `SELECT * FROM delta_table_live WHERE pond_id = ...`
   over the entire pond.  For a steady-state pond with many
   partitions, this is one full-table scan per commit.  Acceptable
   for D5.7a — verify correctness first, optimize later.  Future
   optimization: scan only the partitions touched by this commit
   (we know them from the Delta commit's Add/Remove file actions),
   re-compute their checksums, MERGE with the previous DataCommitted
   record's `partition_checksums` for unchanged partitions.  Defer.

3. **Existing tests calling `record_data_committed` directly.**
   Grep first.  See `crates/steward/src/control_table.rs:821` and
   `crates/steward/src/ship.rs:787` — both look like test-only
   fixtures.  They'll need the new `PartitionChecksums::new()` arg.

4. **`SyncStewardError` → `StewardError` adapter.**  Reuse the
   existing `adapt_err` / `map_err` pattern in
   `crates/steward/src/remote_adapter.rs:13-25`.

## 3. D5.7b — Cross-pond import UX surface

### What's blocked today

`crates/steward/src/remote_adapter.rs:358-368` is the gate:

```rust
let local_pond_id = self.store_id();
if pond_id != local_pond_id {
    return Err(SyncStewardError::Adapter(Box::new(std::io::Error::other(
        format!(
            "duckpond apply_pulled_bundle: cross-pond import is not yet wired \
             through (bundle pond_id {} != local pond_id {}); see docs/d5-resume.md \
             section D5.7",
            pond_id, local_pond_id
        ),
    ))));
}
```

The partition layout (D5.1) already supports it: foreign bundles land
in `pond_id=<foreign_uuid>/part_id=<uuid>/...` directories in the local
Delta table.  The DataCommitted row would record under the foreign
pond_id in the control table.  D5.5's `compute_live_checksums(foreign)`
already returns the foreign data's checksums correctly (test
`ship_compute_live_checksums_unknown_pond_id_is_empty` covers the
empty case; a non-empty foreign import would behave symmetrically).

So the **mechanical** unblock is: delete the guard.  Make
`apply_pulled_bundle` accept any bundle pond_id, store it in the
filesystem under the bundle's `pond_id=...` directory, and write a
DataCommitted record under the bundle's pond_id in the control table.

But that's only half.  The UX questions are:

### Open UX questions

1. **How does an operator declare "I want to import from this remote"?**

   Today `pond remote add <name> <url> --mode=pull` already sets up a
   pull-mode attachment.  But pull on a remote whose `store_id`
   matches the local pond is "mirror mode" (replica restart); pull
   on a remote whose `store_id` differs is "cross-pond import".  The
   distinction is implicit in the store_id comparison.

   **Decision A**: Make the distinction explicit at attach time.  Add
   a `--import` flag to `pond remote add`, or a new verb `pond
   remote attach-import <name> <url> [--mount /imports/<name>]`.

   **Decision B**: Make it implicit.  `pond remote add ... --mode=pull`
   does whichever based on store_id.  Cleaner CLI, harder to reason
   about.

   The sandbox prototype (`crates/sync-tests/tests/cross_pond.rs`)
   doesn't have a CLI; it's pure API.  See lines 66-77 there for the
   pattern: "A pulls from B's remote".  No special verb; the foreign
   store_id is read from the remote and used as the pond_id under
   which to record bundles.

2. **Where does foreign data appear in the filesystem namespace?**

   Two models:
   - **Path-based mount**: foreign data appears under a mount point
     like `/imports/<name>/...`, mirroring the foreign pond's
     filesystem hierarchy.  Requires the filesystem read path to
     know about mounts.
   - **No mount, just data**: foreign data lives at its original
     paths (e.g., if pond B wrote `/sensors/temperature.series`, then
     pond A's import has that data at `/sensors/temperature.series`).
     Collisions with local data are resolved by... pond_id?  Path
     prefix?  Refusal?

   The sandbox model is "no mount" — the foreign data lives in the
   foreign pond_id's namespace, isolated from local data by pond_id
   alone.  But sync_steward is k/v (partition, key); it has no
   hierarchical paths.  Duckpond does.  The path/pond_id collision
   question is genuinely open and needs a design decision.

3. **How does `pond cat /path` decide which pond_id's content to read?**

   Today `pond cat` reads from the local pond_id implicitly.  If
   foreign data sits at the same path, we need either:
   - A flag: `pond cat --pond=<name> /path`.
   - A prefix convention: `pond cat /imports/<name>/path` strips the
     prefix and reads from foreign pond_id.
   - A separate verb: `pond import-cat <name> /path`.

   The path-based mount model makes this implicit and ergonomic.

4. **What happens at `pond remote remove` if foreign data has been
   imported?**

   - Auto-delete the foreign data (call
     `restart_pond_from_compact(drop=true)` or equivalent).
   - Refuse and require explicit `pond unmount` / `pond purge`.
   - Leave the data but stop pulling (decoupled lifetime).

5. **Recursive imports?**  If A imports B, and B imports C, does A
   see C's data?  The push filter (D5.3) says no — A's push to its
   own remote excludes foreign rows, so A's downstream consumers
   don't see B or C through A.  But what about A's own
   `pond list /` — does it surface C?  Currently no, because A's
   pull of B's remote only gets B's own bundles (B's push also
   excludes foreign rows).  This is the intended layered model:
   each import is an explicit declaration.

### Recommended discussion approach

Before any code changes for D5.7b, sit with the user and walk
through:

- The sandbox `cross_pond.rs` test top-to-bottom (lines 42-160).
  This is the pure API behavior we want to mirror in duckpond.
- The four open UX questions above, in order.  Get explicit
  decisions on each.
- The minimal first slice.  Probably: import works, foreign data is
  accessible by pond_id but no fancy mount UX yet.  Ship that as
  D5.7b1.  Add `--mount` and friends as D5.7b2.

## 7. Architecture reminders (so you don't re-derive these)

### Layering (bottom → top, unchanged from D4)

```
cmd          CLI commands (pond init/list/cat/remote/push/pull/...)
sync-remote  Remote (Delta-bundle pipeline) + RemoteSteward trait
steward      Ship + ControlTable + ShipRemoteSteward (adapter)
tlogfs       Delta Lake persistence, OpLog, DataFusion sessions
tinyfs       Pure filesystem abstractions (FS, WD, Node, ...)
```

`sync-remote` is **agnostic** of duckpond's specific schemas.  It
operates against the `RemoteSteward` trait.  `ShipRemoteSteward` (in
`crates/steward/src/remote_adapter.rs`) is the duckpond implementation;
sandbox has its own at `crates/sync-steward/src/steward.rs`.

### Where transactions live (unchanged from D4)

- `pond` CLI = one process = one transaction (enforced by panic if
  a second `TransactionGuard` is created).
- Inside that transaction: pass `&mut tx` to helpers.  **Never** call
  `OpLogPersistence::open` or `persistence.begin()` again.
- Auto-push: post-commit hook fires inside the steward after the
  transaction commits.  Scans `/sys/remotes/*` and pushes to each
  `push|both`-mode remote.  Failures log but don't fail the user's
  write.

### Where remotes live (unchanged from D4)

- **Config**: `/sys/remotes/<name>` as YAML files (in the
  filesystem, inside the steward transaction).  Schema is
  `RemoteAttachment` from
  `crates/steward/src/remote_config.rs:32-55` (url, region, S3
  credentials, allow_http).
- **Mode**: `remote_mode:<name>` in the control table's raw_config
  map.  Schema is `RemoteMode { Push, Pull, Both }`.
- **Per-remote watermarks**: `last_pushed_seq:<url>` and
  `last_pulled_seq:<url>` in the control table's raw_config map
  (keyed by URL, not name, so remote rename works).

The path constants `SYS_DIR = "/sys"`, `SYS_REMOTES_DIR =
"/sys/remotes"`, and the key prefix `REMOTE_MODE_PREFIX =
"remote_mode:"` are exported from `crates/steward/src/lib.rs`.

### Where pond_id lives now (after D5.1 + D5.2)

- **Canonical**: the data Delta table's bootstrap row (pond_id is in
  the row content AND the row's partition values).
- **Cache**: control-table Setting `pond_id` (set by
  `Ship::create_pond_with_metadata`, kept in sync; auto-healed on
  open if it disagrees with the data table).
- **Per-row**: every `OplogEntry` row's `pond_id` column AND the
  partition value of the parquet file the row lives in.
- **Partition layout**: outer `pond_id=<uuid>`, inner
  `part_id=<uuid>`.

`OpLogPersistence::peek_pond_id` reads the canonical value from
Delta partition values without needing a DataFusion session — that
makes it cheap and usable from `open` paths.

### How D5.5's compute_live_checksums works (critical for D5.7a)

`crates/steward/src/remote_adapter.rs:511-547` (full implementation
in the D5.5 commit `ccecc54a`).  Brief recap:

```rust
async fn compute_live_checksums(&self, pond_id: Uuid)
    -> StewardResult<PartitionChecksums>
{
    let table = self.ship.data_persistence().table().clone();
    let ctx = SessionContext::new();
    ctx.register_table("delta_table_live", Arc::new(table))?;

    let sql = format!(
        "SELECT * FROM delta_table_live WHERE pond_id = '{}' \
         ORDER BY part_id, node_id, version",
        pond_id
    );
    let batches = ctx.sql(&sql).await?.collect().await?;

    let mut by_partition = HashMap::<String, Vec<(String, [u8;32])>>::new();
    for batch in batches {
        let rows: Vec<OplogEntry> = serde_arrow::from_record_batch(&batch)?;
        for row in rows {
            let key = format!("{}/{}", row.node_id, row.version);
            let part = row.part_id.to_string();
            let canonical = serde_json::to_vec(&row)?;
            let digest = *blake3::hash(&canonical).as_bytes();
            by_partition.entry(part).or_default().push((key, digest));
        }
    }

    let strategy = Merkle::new();
    let mut out = PartitionChecksums::new();
    for (partition, kv) in by_partition {
        let leaves: Vec<Leaf<'_>> = kv.iter()
            .map(|(k, v)| Leaf { key: k.as_str(), value_blake3: v })
            .collect();
        out.insert(partition, strategy.compute(&leaves));
    }
    Ok(out)
}
```

Important properties:
- **Fresh `SessionContext` per call** — works whether or not a
  transaction is active.  This is why D5.7a can call it from the
  post-commit hook in `guard.rs`.
- **Leaf canonicalization**: `blake3(serde_json::to_vec(&row))` with
  the leaf key `"<node_id>/<version>"`.  Stable across writers and
  re-serializations.  All OplogEntry fields contribute (including
  pond_id, part_id, txn_seq), so foreign-pond rows have the same
  checksum on producer and consumer.
- **Merkle fold** over sorted leaves → 32-byte per-partition
  checksum.  Same strategy sync_store uses.
- **No row-level tombstones in tlogfs** — versions are new rows, so
  the SQL is straight `WHERE pond_id = '...' ORDER BY part_id,
  node_id, version` with no ROW_NUMBER dedup.

D5.7a's `record_data_committed` call must use this exact function to
guarantee that `live == recorded` for the bundle that was just
written.

### Why the doc's original sketch was wrong

Section 2 proposed:
```rust
let cs = self
    .ship                                  // <-- doesn't exist
    .partition_checksums_at(/* ... */)
    .await?;
```

`StewardTransactionGuard` does NOT hold a `&Ship`.  It holds
`data_tx: Option<TransactionGuard<'a>>` and `control_table: &'a mut
ControlTable`.  The `data_tx`'s `TransactionGuard` internally holds
`&'a mut OpLogPersistence` (the data persistence).

The challenge: `data_tx.commit().await` consumed the `data_tx`,
which released the `&mut` to the persistence.  After commit we
need the persistence again (to read the post-commit Delta state
for the checksum snapshot), but we have no other path to it
without a refactor.

### The actual approach

Three options were considered:

A. **Reopen persistence from disk** — too heavy (a fresh
   `OpLogPersistence::open` does a Delta log scan).
B. **Hold a duplicate `&mut` in the guard** — forbidden by borrow
   checker.
C. **Return the persistence ref from `commit`** — minimal: only
   one caller, `StewardTransactionGuard::commit`.  Chose this.

Implementation hazard: `TransactionGuard` implements `Drop`, so
the obvious `let Self { inner, persistence, .. } = self;`
destructure fails with E0509.  The idiomatic safe-Rust workaround
is `ManuallyDrop` + `ptr::read`:

```rust
#[allow(unsafe_code)]
let (inner, persistence, metadata, is_write) = {
    let this = std::mem::ManuallyDrop::new(self);
    // SAFETY: each field is read exactly once and `this` is never
    // accessed after the reads, so we cannot observe a moved-out
    // field through `this`.
    unsafe {
        (
            std::ptr::read(&this.inner),
            std::ptr::read(&this.persistence),
            std::ptr::read(&this.metadata),
            this.is_write,
        )
    }
};
```

The workspace lint `unsafe_code = "deny"` requires the
`#[allow(unsafe_code)]` at the destructure block.  The
`TransactionGuard`'s Drop impl is a no-op once `persistence.state`
and `persistence.fs` are `None`, which `commit()` sets before
returning.

### Limit of the fix

D5.7a closes the "RECORDED is always empty for native writes"
bug.  It DOES NOT make cross-pond push/pull/verify symmetric.

The regression test was originally drafted to push from src, pull
into dst, and verify both sides.  The dst verify failed even
after D5.7a, because:

- src's `compute_live_checksums` returns checksums for ALL of
  src's data table rows where `pond_id = src_pond_id`, including
  the seq=1 pond_init rows that D5.6 refuses to push.
- dst's data table starts empty.  After pulling seq=2, dst has
  ONLY the seq=2 bundle's rows -- no seq=1 pond_init.
- Therefore src's "live" and dst's "live" (each computed against
  its own table) cannot agree on the root partition.

Resolving that is a separate design question (push self-contained
bundles?  exclude unpushable rows from src's checksum?  give dst
a deterministic pond_init?).  It is OUT OF SCOPE for D5.7a and
should be a separate phase if pursued.  The regression test was
narrowed to the src-side native-write verify, which is exactly
the originally-broken path.
