# Caspar Water — Agent Instructions

This directory manages a DuckPond instance running on an amd64 Linux
machine (`linux.local`) that collects and visualizes OTelJSON telemetry
from a community water system (well depth, tank level, system pressure,
chlorine level, pH).

## Architecture

- **linux.local** is an amd64 machine running `podman` with the
  `duckpond:latest-amd64` container image.
- An OTel Collector on the device writes OTelJSON logfiles to `/home/data/`.
- `pond-remote.sh` wraps every `pond` CLI call via SSH + podman on the
  remote host.
- Config files (YAML, markdown templates) live in this directory and are
  `scp`'d to the remote at `/home/jmacd/water-config/`.

## Data: Sample vs Full

There are two data sources — knowing the difference is critical:

| Data | File(s) | Size | Coverage | Used by |
|------|---------|------|----------|---------|
| **Sample** | `casparwater-sample.json` | ~100 MB, 1 file | ~10 days | `render.sh` (hostmount) |
| **Full** | `data/casparwater*.json` | ~9 GB, ~96 rotated logfiles | 2022–2026 | Local & remote pond workflows |

- **Sample data** is committed alongside the scripts. It exists so you can
  quickly test blog content and site template changes without waiting for
  a full ingest. It only covers ~10 days of telemetry.
- **Full data** lives in `data/` (rsynced from `linux.local:/home/data/`).
  To see the complete historical dataset you must use one of the pond
  workflows (local or remote) which ingest all rotated logfiles.

## Scripts (run from this directory)

There are three workflows, each suited to a different task:

### 1. Hostmount Workflow — Sample Data (fastest, for template/blog dev)

Uses only `casparwater-sample.json` (~10 days). No pond, no ingest.
Use this when iterating on site templates, blog content, or CSS — you
don't need historical data for that.

| Script | Purpose |
|--------|---------|
| `render.sh` | Build site from sample data using hostmount overlay, output to `dist/` |
| `render-data.sh` | (Legacy) Same idea but builds a temporary pond |

```bash
./render.sh             # builds site, opens in browser on :4175
./render.sh --no-open   # build only, output in dist/
```

### 2. Local Pond Workflow — Full Data (runs via cargo on this machine)

Rsyncs all logfiles from `linux.local`, ingests into a local pond in
`./pond/`, and generates the site with the complete dataset.

| Script | Purpose | When to use |
|--------|---------|-------------|
| `setup-local.sh` | Rsync data, wipe & init pond, install factories | First time or full reset |
| `update-local.sh` | Recreate factory nodes with `--overwrite` | After editing YAML or site templates |
| `run-local.sh` | Rsync latest data, ingest into local pond | To pull in new telemetry data |
| `generate-local.sh` | Run sitegen, output to `export/` | To build the static site |

```bash
# Full reset (destructive — wipes ./pond/)
./setup-local.sh        # rsync data + init pond + install factories
./run-local.sh          # ingest all logfiles (slow — large dataset)
./generate-local.sh     # build site → export/

# Preview
cd export && python3 -m http.server 4177
```

Config-only update (pond already exists):

```bash
./update-local.sh       # recreate factory nodes
./generate-local.sh     # rebuild site
```

### 3. Remote Pond Workflow — Full Data (runs on linux.local via podman)

Runs everything on the remote amd64 host. The container image is built
locally and pushed via SSH.

| Script | Purpose | When to use |
|--------|---------|-------------|
| `build-remote.sh` | Build amd64 container image, push to linux.local | After code changes to duckpond |
| `setup-remote.sh` | Wipe pond volume, init, copy templates, install factories | First time or full reset |
| `update-remote.sh` | Push updated configs, recreate factories with `--overwrite` | After editing YAML or site templates |
| `run-remote.sh` | Ingest new logfile data (`pond run /etc/ingest`) | To pull in new telemetry data |
| `generate-remote.sh` | Run sitegen, rsync built site to `export/` | To build the static site |

```bash
# Full reset
./build-remote.sh       # build + push amd64 image (only if code changed)
./setup-remote.sh       # wipe pond volume, init, install factories
./run-remote.sh         # ingest logfiles
./generate-remote.sh    # build site, rsync → export/

# Preview
npx vite export --port 4177 --open
```

Config-only update (pond already exists):

```bash
./update-remote.sh     # push configs + recreate factories
./run-remote.sh        # re-ingest if needed
./generate-remote.sh   # rebuild site
```

## Choosing a Workflow

| I want to... | Use |
|--------------|-----|
| Iterate on blog content or site templates quickly | `render.sh` (sample data, ~3 min) |
| See the full historical dataset locally | Local pond workflow (full data, longer) |
| Deploy on the production host | Remote pond workflow |
| Test after duckpond code changes (remote) | `build-remote.sh` → remote workflow |
| Test after duckpond code changes (local) | Local workflow (uses `cargo run`) |

## Config Files

| File | Used by | Data source | Description |
|------|---------|-------------|-------------|
| `site.yaml` | Hostmount, local pond | — | Sitegen config (references host paths for templates) |
| `site-remote.yaml` | Remote pond | — | Sitegen config (references pond paths for templates) |
| `reduce.yaml` | Hostmount (`render.sh`) | Sample only | Temporal-reduce reading `oteljson:///casparwater-sample.json` |
| `reduce-remote.yaml` | Local & remote pond | Full data | Temporal-reduce reading `oteljson:///ingest/casparwater*.json` |
| `ingest.yaml` | Remote pond | Full data | Logfile-ingest config (reads from `/data/`) |

## Key Details

- **amd64, not arm64.** The container image is `duckpond:latest-amd64`.
  `build-remote.sh` uses `docker buildx build --platform linux/amd64`.
- **Large dataset.** The full dataset is ~9 GB across ~96 rotated logfiles
  (2022–2026). Ingest and sitegen take significantly longer than for the
  septic station. Watch memory on linux.local.
- `setup-remote.sh` and `setup-local.sh` are destructive — they wipe the
  pond volume / directory. Only use for a full reset.
- The ingest factory reads from `/data` (host mount) and writes to
  `/ingest/` in the pond.
- Temporal-reduce (`/reduced`) is a dynamic factory — computed on read,
  no explicit run step needed.
- Sitegen exports to `/output` inside the container (remote), `export/`
  (local pond), or `dist/` (hostmount).
- The generated site includes 5 metric groups (well-depth, tank-level,
  system-pressure, chlorine-level, pH) at 5 resolutions (1m, 10m, 1h,
  6h, 1d).

## Troubleshooting

- **Only ~10 days of data visible** → you used `render.sh` which reads
  only the sample file. Use the local or remote pond workflow for the
  full dataset.
- **"Not a Delta table"** on `pond list` → pond isn't initialized; run
  `setup-remote.sh` or `setup-local.sh`.
- **"NotFound: /site" or "/content"** during setup → `mkdir -p /etc` or
  directory creation is missing before `pond copy`.
- **No rotated logfiles** → only the active `casparwater.json` exists;
  the ingest factory handles it.
- **OOM on linux.local** → the water dataset is large; sitegen may need
  significant memory for temporal-reduce aggregations across all 5 metrics
  × 5 resolutions.
- **Build architecture mismatch** → ensure `build-remote.sh` targets
  `linux/amd64`, not `linux/arm64`.
